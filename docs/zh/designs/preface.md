# 前言

简单讲讲一些 Hunea 的设计方面的考虑。

不过这些设计可能在未来也会进行变更，不过当前的话应该是比较稳定的。如有变化后续也会进行文档的调整的。

如果你主要关心操作路径，可以先看 [斜杠菜单](/guide/fun/menu/preface)。

## 为什么使用 altscreen

Hunea 的主界面跑在终端的 **alternate screen**（常说 altscreen / 备用屏幕缓冲）里。

简单来说，进入 altscreen 之后，应用也就是 Hunea 拿到的是一块**独立的全屏画布**：帧刷新、全屏面板、覆盖层、状态行，都可以按自己的布局画；退出时再离开 altscreen，把终端交还给原来的主屏幕。这样主会话不会被 TUI 的中间帧、清屏和局部重绘弄脏，退出路径也更干净。

1. **全屏模态层需要一整块可控区域**  
   [`/tree`](/guide/fun/menu/tree)、[`/copy`](/guide/fun/menu/copy)、[`/prompt`](/guide/fun/menu/prompt)、会话恢复列表这类界面，不是贴在底部的小浮窗，而是会接管整屏的列表、预览和分页。altscreen 让这些层可以直接占满终端，而不是和 shell 的 scrollback 抢同一条历史。
2. **输入框可以跟着内容一起向下滚动，而不是固定在底部**  
   主文档把 transcript 和 composer 当成同一份可滚动文档来排。composer 可以无界增高，不会在输入框内部再开一层滚动。这种布局在主屏幕 + 固定底栏的模型里很难做干净；在 altscreen 里则可以按帧重算。
3. **退出后的「回看」可以做成显式选择**  
   altscreen 的代价是：会话进行中，终端模拟器自己的 scrollback 里通常看不到 TUI 内部帧。Hunea 不把「历史一定要堆在终端滚动缓冲里」当成默认，而是提供可选的退出回放：配置里的 `print_transcript_on_exit` 为 `true` 时，退出后会把当前对话以 terminal replay 形式打回主屏幕 scrollback。默认是关闭的，因为很多人只是结束一次会话，并不想在 shell 里再刷出一整段 transcript。

其实一开始我也考虑想做 `Codex CLI` 那样的底部重绘，然后内容都进终端模拟器的原生 scrollback 区域。因为好处很多：

1. 鼠标选区、复制等都是终端模拟器原生的功能，操作起来非常自然而且符合用户的使用习惯
2. 滚动流畅。因为滚动什么的终端模拟器原生就处理好了，所以非常自然，包括那个滚动加速度等优化都很舒服
3. 一时想不出来了……

但是后面还是选择没有做，原因和很简单：

1. 工作量太大了
2. 如果没有处理好，就会有各种奇怪的问题
3. 有时候会出现一些意外的问题

哪怕是成熟如 `Claude Code` 都提供了 altscreen 的切换选择，好巧不巧的是，前段时间我在使用 `Claude Code` 的时候，就碰到了问题：

![index-1](/assets/designs/index1.png)

我的 `Claude Code` 版本是 `2.1.21x`，已经很新了，但是还是会有类似的问题出现。`Codex CLI` 好些，但是也会出现问题。而且，在部分终端模拟器中，我观察到还会有 scrollback 内容丢失的情况，我猜测是内容推入 scrollback 后，因为终端模拟器或者其他的兼容性原因，导致一段很长的内容中会丢失中间部分内容。

不过 `Codex CLI` 在大约几个月前的版本更新中加入了一个新的功能，忘记叫什么名字了（刚刚看了一下自己的 v0.144.1 版本的 `Codex CLI` 也没有看到这个功能了），可以实现在终端模拟器显示窗口大小变化后重建内容。也算是变相解决了这类问题，我只要改变一下我的终端模拟器显示窗口大小就好了……

所以我还是选择了 altscreen 的方式，然后尽可能的做各种优化，使其性能和效果上更接近于底部重绘的体验。现在的性能倒还是不错的。未来也会继续打磨。

## 为什么有的界面能用鼠标选区复制，有的不能

终端里的「鼠标」其实有两套语义，Hunea 会按当前界面在两套之间切换：

| 模式（概念） | 鼠标点选 / 拖拽归谁 | 典型场景 |
| --- | --- | --- |
| **应用捕获** | Hunea 自己处理点击、拖拽选区、列表点选 | 主对话区、composer，以及需要点行切换的列表 |
| **终端原生** | 把选区能力还给终端模拟器 | 偏「阅读 / 预览」的全屏层 |

默认主界面走**应用捕获**。原因很直接：

- 需要鼠标点一下，就把 composer 光标落到对应位置
- 需要在 transcript 上拖出一块**语义选区**，复制的是正文，而不是左侧装饰、缩进、状态行边距
- 需要在列表里点行移动光标，但不误触成「终端选中半行乱码再粘贴」

所以主界面的选区是 Hunea 自己算的：命中范围和复制范围是分开的，左边的提示符、视觉缩进可以更好点中，但复制时不会把这些装饰一起带上。选区结束后是否自动写入剪贴板，由 `copy_on_mouse_selection_release` 控制，默认关闭；中键也可以触发复制当前选区。消息级整段拷贝则走 [`/copy`](/guide/fun/menu/copy)，和屏幕拖选是两条路径。

但全屏层不能一刀切。Hunea 按「这一层此刻更需要点选，还是更需要读和拷」来切鼠标模式：

- **需要点行、需要滚轮当导航**的列表（例如 `/tree`、`/copy`、消息历史、`/prompt` 的列表态）  
  继续由应用捕获鼠标，并打开 alternate scroll，让滚轮稳定变成翻页/移动，而不是被终端占用。
- **已经进入内容预览**（例如树节点预览、copy 预览、prompt 预览）  
  关闭鼠标捕获，把**原生选区**还给终端模拟器，同时保留 alternate scroll，滚轮仍可翻页。  
  也就是说：预览页更像「打开了一份只读文本」，你可以直接用终端习惯去划词复制。
- **本身就以阅读为主**的层（例如 transcript overlay、会话预览）  
  同样偏向原生选区 + alternate scroll。

一句话总结：

> **要交互时，鼠标归 Hunea；要阅读和划词时，比如 space 预览界面和 `Ctrl + t` 进入的详情界面，鼠标归终端。**

这就是你会感觉「有的界面能像普通终端一样拖选，有的界面拖不动、却能点选列表」的原因。

## 其他一些设计和取舍

### 主文档是「一份可滚动文档」，不是「底栏输入框 + 上方面板」

transcript 和 composer 共享同一套文档布局。Hunea 没有把「底部永远固定输入框区域」作为首选，因为那会稳定地占用掉阅读高度，长输入还容易进入输入框内部滚动路径。

哪怕后续提供固定底栏等更多方案，也是可切换选项，而不是替换掉现在这套。

### 配置默认偏克制

例如：

- `print_transcript_on_exit` 默认 `false`：退出不刷屏
- `copy_on_mouse_selection_release` 默认 `false`：选区完成不自动进剪贴板，避免误触
- reasoning 默认不展开长文：需要时再进 transcript overlay 细看

能开箱即用，但不会默认替你制造噪音。
